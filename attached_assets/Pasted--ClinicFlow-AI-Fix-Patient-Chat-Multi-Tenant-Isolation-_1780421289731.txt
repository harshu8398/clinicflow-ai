# ClinicFlow AI – Fix Patient Chat Multi-Tenant Isolation

## IMPORTANT

DO NOT modify:

* Authentication system
* Admin login
* Authorization middleware
* Dashboard security
* UI design
* Routing structure (except validation fix)

This is NOT an auth change.

This is a data isolation fix.

---

# 🚨 CURRENT CRITICAL ISSUE

Patient chat URLs allow cross-clinic access:

Example:

/chat/1
/chat/2
/chat/3

Currently, any clinic_id can be accessed directly by changing URL.

This breaks multi-tenant security.

---

# 🎯 REQUIRED BEHAVIOR

Each chat must be strictly bound to clinic_id.

A patient accessing:

/chat/3

must ONLY see data for clinic_id = 3

and NEVER any other clinic data.

---

# 🔐 REQUIRED FIXES

## 1. Backend Validation (MANDATORY)

Every chat API / query must enforce:

clinic_id match is REQUIRED

Example fix:

```sql
SELECT *
FROM messages
WHERE session_id = ?
AND clinic_id = ?
```

---

## 2. Session Binding

When session is created:

session must store clinic_id

Example:

session_id → clinic_id binding

A session created for clinic 3:

CANNOT access clinic 1 or 2

---

## 3. Route Protection (Patient Chat)

Route:

/chat/<clinic_id>

must validate:

IF clinic_id is not valid OR mismatch occurs:
→ reject request OR redirect to safe page

---

## 4. API Security Rule

ALL chat-related endpoints MUST enforce:

* clinic_id match
* session_id match
* no cross-clinic data leakage

---

## 5. Data Isolation Rule

Ensure:

Clinic A data = ONLY Clinic A

Clinic B data = ONLY Clinic B

NO SHARED ACCESS EVER
